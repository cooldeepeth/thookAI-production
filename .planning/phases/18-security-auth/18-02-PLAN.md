    Create `backend/tests/security/test_oauth_security.py` with these test classes.

    Reuse the helper pattern from test_oauth_flows.py (_make_platforms_app, _make_google_app, _mock_httpx_client). Import or re-define as needed.

    **class TestOAuthStateForgery** (per SEC-02 — state parameter validation):
    - `test_linkedin_callback_forged_state_redirects_to_error`: Call /callback/linkedin?code=x&state=FORGED_STATE. Patch db.oauth_states.find_one_and_delete to return None (state not found). Assert 302 redirect with "error" in location URL.
    - `test_twitter_callback_forged_state_redirects_to_error`: Same pattern for /callback/x?code=x&state=FORGED_STATE. Assert redirect to error.
    - `test_instagram_callback_forged_state_redirects_to_error`: Same for /callback/instagram?code=x&state=FORGED_STATE. Assert redirect to error.
    - `test_google_callback_token_exchange_failure_redirects_to_error`: Patch _oauth.google.authorize_access_token to raise Exception. Assert redirect to /auth?error=oauth_failed.
    - `test_linkedin_callback_missing_state_param_returns_error`: Call /callback/linkedin?code=x (no state param at all). Assert error response (400 or redirect to error).
    - `test_twitter_callback_missing_code_param_returns_error`: Call /callback/x?state=valid (no code param). Assert error response.
    - `test_linkedin_callback_expired_state_rejected`: Create oauth_state_doc with expires_at in the past, but since find_one_and_delete checks state string match (not expiry in the query), the state may still be found. Test that if the state doc has expired_at < now, the callback still processes OR add a note if the implementation doesn't check expiry. Verify the state is consumed (deleted after use) so replays fail.
    - `test_state_consumed_after_use_replay_fails`: First callback with valid state succeeds (find_one_and_delete returns doc). Second callback with same state fails (find_one_and_delete returns None because already deleted). Assert second call redirects to error.

    **class TestOAuthTokenEncryption** (per SEC-02 — token storage encryption):
    - `test_linkedin_token_stored_encrypted_not_plaintext`: Capture the doc passed to db.platform_tokens.update_one after LinkedIn callback. Assert access_token value != the plaintext token from httpx response.
    - `test_twitter_token_stored_encrypted_not_plaintext`: Same for X/Twitter callback.
    - `test_instagram_token_stored_encrypted_not_plaintext`: Same for Instagram callback.
    - `test_encrypt_decrypt_roundtrip_is_symmetric`: Import _encrypt_token, _decrypt_token from routes.platforms. Assert _decrypt_token(_encrypt_token("test_token")) == "test_token".
    - `test_different_plaintexts_produce_different_ciphertexts`: Two different tokens produce different encrypted values.

    **class TestOAuthPKCEIntegrity** (per SEC-02 — Twitter PKCE):
    - `test_twitter_pkce_code_verifier_sent_in_token_exchange`: After GET /connect/x stores state+code_verifier, callback POST to token endpoint must include code_verifier matching what was stored. (This exists in test_oauth_flows.py but verify it's in the security suite too for SEC-02 coverage.)

    Total: ~15 OAuth security tests.

    **class TestAuthRateLimitBurst** (per SEC-03 — concurrent burst):
    - `test_100_concurrent_auth_requests_trigger_429`: Use asyncio.gather with 100 concurrent calls to RateLimiter.is_rate_limited with key "ip:burst:/api/auth/login" and limit=10. Assert at least 90 of 100 are blocked (is_limited=True). This simulates 100 req/s burst against auth.
    - `test_auth_rate_limit_blocks_at_threshold_10`: Sequential loop: fire exactly 10 requests to /api/auth/login via TestClient (each returns 200), then fire the 11th — assert 429.
    - `test_register_endpoint_has_same_auth_limit`: Fire auth_limit+1 requests to /api/auth/register. Assert the last one returns 429.
    - `test_forgot_password_has_auth_limit`: Fire auth_limit+1 requests to /api/auth/forgot-password. Assert 429.

    **class TestRateLimitPerIP** (per SEC-03 — per-IP independence):
    - `test_different_ips_have_independent_limits`: Client A (X-Forwarded-For: 10.0.0.1) exhausts limit. Client B (X-Forwarded-For: 10.0.0.2) should NOT be blocked. Use TestClient with custom headers.
    - `test_x_forwarded_for_header_used_for_ip`: Fire requests with X-Forwarded-For: "1.2.3.4". Exhaust limit. Fire request WITHOUT header (uses testclient default). Should NOT be blocked (different IP).

    **class TestRateLimitHeaders** (per SEC-03 — response headers):
    - `test_429_response_includes_retry_after`: Exhaust auth limit, assert 429 response has "Retry-After" header with integer value.
    - `test_429_response_body_includes_detail`: Assert 429 response JSON has "detail" key with user-friendly message.
    - `test_200_response_includes_rate_limit_remaining`: First request to any endpoint, assert X-RateLimit-Remaining header is present and numeric.
    - `test_health_endpoint_exempt_from_rate_limiting`: Fire 100+ requests to /api/health, none should return 429 (health is in skip list).
