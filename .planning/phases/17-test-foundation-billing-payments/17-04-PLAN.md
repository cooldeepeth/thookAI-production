    **File 1: test_subscriptions.py (~45 tests for BILL-02)**

    Create `backend/tests/billing/test_subscriptions.py` covering the full subscription state machine:

    **handle_checkout_completed:**
    - Credit purchase checkout → adds credits to user
    - Custom plan checkout → activates subscription, sets tier + credits
    - Missing user_id in metadata → logs error, does not crash
    - Missing checkout type in metadata → handled gracefully

    **handle_subscription_created:**
    - New subscription → writes subscription record to DB with correct fields
    - Subscription with metadata (user_id, plan) → user tier updated
    - Subscription without user_id metadata → logged as warning

    **handle_subscription_updated:**
    - Plan upgrade (starter → pro) → user tier updated, credits adjusted
    - Plan downgrade (pro → starter) → user tier updated at period end
    - Status change to past_due → user flagged

    **handle_subscription_deleted:**
    - Active subscription deleted → user reverted to free/starter tier
    - Credits not removed on cancellation (they're pre-paid)

    **handle_payment_succeeded:**
    - Invoice with subscription → monthly credits refreshed
    - Invoice with credit purchase metadata → credits added
    - Invoice without relevant metadata → handled gracefully

    **handle_payment_failed:**
    - Failed payment → user marked as payment_failed status
    - Subscription status updated in DB

    **cancel_stripe_subscription:**
    - Cancel at period end → calls Stripe API, updates DB
    - Cancel immediately → calls Stripe API, reverts tier
    - User with no subscription → returns error
    - Stripe API error → returns error

    **modify_subscription:**
    - Upgrade credits → updates subscription via Stripe API
    - Downgrade credits → updates subscription
    - Invalid subscription → returns error

    **Route-level:**
    - POST /api/billing/subscription/cancel → 200
    - POST /api/billing/plan/modify → 200
    - GET /api/billing/subscription → returns subscription status
    - POST /api/billing/webhook → processes event (with mock construct_event)

    Use `mongomock_db` for all tests that verify DB state. Use `respx` for tests that exercise Stripe API calls. Use `make_user` factory for test data.
