    **Webhook Signature Verification:**
    - Valid signature → processes event
    - Invalid signature → returns `{"success": False, "error": "Invalid signature"}`
    - Empty payload → returns `{"success": False, "error": "Invalid payload"}`
    - Stripe not configured → returns error
    - Webhook secret not configured → returns error

    **Webhook Idempotency (expanding on Plan 02 P0 tests):**
    - Duplicate `checkout.session.completed` → duplicate=True (covered by Plan 02, but verify again in integration)
    - Duplicate `customer.subscription.created` → duplicate=True, subscription not double-created
    - Duplicate `customer.subscription.updated` → duplicate=True, tier not double-updated
    - Duplicate `customer.subscription.deleted` → duplicate=True, tier not double-reverted
    - Duplicate `invoice.payment_succeeded` → duplicate=True, credits not double-refreshed
    - Duplicate `invoice.payment_failed` → duplicate=True, status not double-set
    - Different event IDs same type → both processed (no false positive dedup)
    - Event ID stored in stripe_events with TTL-compatible timestamp

    **Webhook Event Routing:**
    - `checkout.session.completed` → calls handle_checkout_completed
    - `customer.subscription.created` → calls handle_subscription_created
    - `customer.subscription.updated` → calls handle_subscription_updated
    - `customer.subscription.deleted` → calls handle_subscription_deleted
    - `invoice.payment_succeeded` → calls handle_payment_succeeded
    - `invoice.payment_failed` → calls handle_payment_failed
    - `unknown.event.type` → logs info, returns success (graceful ignore)

    **Credit Atomicity Under Concurrent Deductions (BILL-03):**
    Use mongomock_db for all atomicity tests.

    - `deduct_credits` happy path → reduces balance correctly via `find_one_and_update`
    - `deduct_credits` insufficient credits → returns error, balance unchanged (verified by re-reading DB)
    - Two concurrent `deduct_credits(CONTENT_CREATE)` on user with exactly 20 credits → first succeeds (balance=10), second fails (insufficient) — NOT both succeed producing -10
    - Three concurrent `deduct_credits(CONTENT_CREATE)` on user with 25 credits → max 2 succeed, at least 1 fails
    - `deduct_credits` with exact balance (credits == operation cost) → succeeds, balance = 0
    - `deduct_credits` records transaction document
    - `add_credits` + `deduct_credits` interleaved → final balance is correct (100 + 50 - 10 = 140)
    - Starter tier cap enforcement: more than daily limit video generations → blocked
    - Starter tier cap enforcement: more than daily limit carousel generations → blocked

    **Retry Behavior:**
    - Webhook that throws during handler → returns `{"success": False, "error": "..."}` and event is NOT marked in stripe_events (allowing retry)
    - After handler error, re-delivery of same event → processes again (error was not recorded as success)

    For each test, use `mongomock_db` fixture and patch both `services.stripe_service.db` and `services.credits.db` to the same mongomock instance to simulate shared database state.

    Mock `stripe.Webhook.construct_event` to return controlled event payloads. Do NOT call real Stripe API.
